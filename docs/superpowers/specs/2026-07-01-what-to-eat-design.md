# 「今天吃什么」产品设计文档

> 版本 1.0 | 2026-07-01 | 作者: SS & Claw

---

## 1. 产品概述

### 1.1 一句话描述

用最少的信息输入，告诉用户现在能做什么菜。

### 1.2 核心价值主张

下班回家、打开冰箱、不知道做什么——拍照或说一句「我有鸡蛋、西红柿、青椒」，立刻得到 2-3 个可行菜谱，附带做法步骤。

### 1.3 差异化定位

| 维度 | 传统食谱 App | 「今天吃什么」 |
|------|------------|------------|
| 使用方式 | 搜菜名 → 看菜谱 → 去买菜 | 说/拍你有什么 → 告诉你能做什么 |
| 决策起点 | 用户先想好吃什么 | 系统根据现有条件推荐 |
| 用户心智 | 「我要学做XX菜」 | 「我懒得想，你帮我决定」 |

### 1.4 目标用户

独居或小家庭的日常做饭人。典型画像：

> 小张，独居，上海上班族。晚上 7 点半回到家，打开冰箱看到还剩两个番茄、三个鸡蛋、一把青菜。打开小程序拍照，系统识别出这些食材，结合他「15 分钟以内」「中式家常菜」的偏好，推荐：番茄炒蛋、上汤青菜。他选了番茄炒蛋，按步骤做完，15 分钟吃上饭。

## 2. MVP 功能范围

### 2.1 三个核心模块

**模块一：食材输入（入口）**
- 📸 拍照识别：拍冰箱/食材 → AI 视觉识别 → 自动列出食材清单
- 🎤 语音输入：「我有鸡蛋、番茄、青椒、豆腐」
- ✍️ 手动勾选：常用食材网格，快速选择

**模块二：智能推荐（核心引擎）**
- 基于食材 × 用户偏好（口味、时间、难度）→ LLM 推荐 2-3 道菜
- 排序逻辑：食材匹配度 > 临期提醒 > 用户历史偏好 > 热门度
- 每道菜展示：菜名、时间、难度、食材匹配状态（全齐 / 缺什么）

**模块三：做菜引导（出口）**
- 分步骤展示，每步图文
- 语音播报（TTS）下一步（方便手湿的情况）
- 关键步骤一键计时
- 做完后收集简单反馈（😋好吃 / 😐一般 / 👎不做了）

### 2.2 用户主路径

```
打开小程序 → 输入食材（拍/说/选）→ 看推荐 → 选一道菜 → 跟着步骤做
```

目标：从打开到开始做菜，**30 秒以内**。

### 2.3 MVP 明确不做

- ❌ 食材库存持久化管理（每次都是「现在有什么」）
- ❌ 购物清单
- ❌ 社区/分享
- ❌ 营养分析
- ❌ 周餐单规划

## 3. 平台 & 技术架构

### 3.1 平台

微信小程序（零安装、获客成本低、厨房场景自然）

### 3.2 架构图

```
┌──────────────────────────────────────────┐
│              微信小程序 (前端)              │
│  ┌─────────┐ ┌──────────┐ ┌───────────┐  │
│  │ 食材输入  │ │ 菜谱推荐  │ │ 做菜引导  │  │
│  │ 拍照/语音│ │ 结果卡片  │ │ 步骤播放  │  │
│  └────┬─────┘ └────┬─────┘ └─────┬─────┘  │
└───────┼────────────┼─────────────┼────────┘
        │            │             │
   ┌────▼────────────▼─────────────▼────┐
   │           API 网关层               │
   │         (云函数 / BFF)             │
   └────┬────────────┬────────────┬─────┘
        │            │            │
   ┌────▼────┐ ┌─────▼──────┐ ┌───▼──────┐
   │ 视觉识别 │ │ 推荐引擎   │ │ 用户数据  │
   │ (AI API)│ │ (LLM+规则) │ │ (DB)     │
   └─────────┘ └────────────┘ └──────────┘
```

### 3.3 组件说明

| 组件 | 职责 | 技术选型 |
|------|------|---------|
| 前端小程序 | UI 交互、拍照/录音、步骤展示 | 微信原生 / Taro / uni-app |
| API 网关 | 鉴权、路由、参数校验 | 云函数 |
| 视觉识别 | 拍照 → 食材名称列表 | 视觉 AI API（通义千问/混元等） |
| 推荐引擎 | 食材 + 偏好 → 菜谱推荐 | LLM 实时生成（无菜谱数据库） |
| 用户数据 | 偏好、历史、反馈 | 关系型存储 |

### 3.4 核心数据流

```
用户拍照 → 图片上传云存储 → 视觉识别 API 返回食材列表
    → 合并用户偏好（口味/时间/难度）
    → 推荐引擎 LLM 生成推荐 + 排序
    → 返回 2-3 个推荐结果
    → 用户选择 → LLM 生成步骤详情 → TTS 语音播报
```

### 3.5 菜谱策略

采用 **LLM 实时生成**（无菜谱数据库）：
- 优点：零内容成本，上线即完整，灵活适配任意食材组合
- 风险控制：通过严格的 prompt 工程约束菜谱质量，关键步骤做格式校验

## 4. 页面结构 & 交互

### 4.1 页面 1：首页 — 食材输入

```
┌─────────────────────────────┐
│    今天想吃什么？            │
│                             │
│   ┌─────────────────────┐   │
│   │     📸 拍一拍冰箱    │   │  ← 主 CTA
│   └─────────────────────┘   │
│                             │
│   🎤 按住说话              │  ← 次入口
│   ✍️ 手动选食材            │  ← 第三入口
│                             │
│   [鸡蛋] [番茄] [青椒] [豆腐]   │  ← 最近常用快捷入口
│                             │
│   ⚙️ 15分钟 · 中式 · 家常   │  ← 偏好设置
└─────────────────────────────┘
```

交互要点：
- 拍照识别中显示骨架屏动画
- 识别结果可编辑（删/改/加）
- 确认食材后自动跳转推荐页

### 4.2 页面 2：推荐结果

展示 2-3 道推荐菜，每道包含：
- 菜名 + 封面图
- 时间 + 难度
- 食材匹配状态（🟢全齐 / 🟡缺XX）
- 「换一批」和「添加更多食材」操作

### 4.3 页面 3：做菜步骤

- 左右滑动切换步骤
- 每步：示意图 + 文字说明
- 🔊 语音播报按钮
- ⏰ 关键步骤一键计时
- 做完后收集反馈

## 5. 异常处理 & 容错

| 场景 | 处理策略 |
|------|---------|
| 拍照识别不出 | 降级到手动输入，保留已识别部分 |
| 识别出非食材 | 过滤 + 用户可手动删除 |
| 食材太少（<3种） | 仍然尝试推荐，提示「食材不多，但也能做」 |
| 食材无法组合 | 逐个推荐单品用法，不强行拼凑 |
| LLM 超时 (>5s) | 骨架屏动画 → 超时降级到本地规则推荐 |
| 步骤缺失/不合理 | 前端校验格式（准备→烹饪→出锅），不通过则重生成 |
| 网络断开 | 已加载步骤可离线查看，新请求提示网络状态 |

### 冷启动

- 默认偏好：时间/口味/难度均为「不限制」
- 前 3 次使用后弹出轻量偏好收集（2 道选择题，非注册页）
- 每次反馈（😋/😐/👎）逐步训练推荐

---

## 6. 非功能需求

- **响应时间**：推荐结果 5s 内返回，超时降级
- **准确率**：食材识别准确率目标 > 85%（MVP 可接受手动修正）
- **可用性**：核心路径离线降级可用（已加载的步骤）
- **合规**：不强制收集个人信息，偏好数据本地优先

---

## 7. 后续迭代方向（非 MVP）

- 食材库存持久化 → 临期提醒 → 减少浪费闭环
- 购物清单自动生成
- 营养分析与健康目标跟踪
- 社区分享 & UGC 菜谱
- 多设备同步（小程序 + Web）
- 智能补货推荐（对接生鲜电商）
